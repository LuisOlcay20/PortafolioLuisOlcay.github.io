Fundamentals of Data Modeling and DAX Basics

Data Model Basics:

Data tables (fact tables): contain business values such as quantities, incomes, pageviews, etc.

Lookup tables (dimension tables): provide descriptive attributes of the model such as customers, products, dates, etc.

These tables are connected through "Primary Keys" and "Foreign Keys". This is where the concept of "Cardinality" (referring to the unique values in a column) and "Filter Flow" (which direction the filter is applied, up or down) come into play.

Data Model Best Practices:

  • Star Schema: 1:*
  • One-way Filter Flow, not bidirectional
  • No redundant tables
  • Create a date table

DAX Basics:

Operators: & (concatenate text), && (AND condition), || (OR condition), in (OR condition within a {list})

Key Functions:

  • Calculate (Expression, [f1], [f2], ...): evaluates an expression under a set of defined filters. This function modifies and overrides its "Filter Context" with the ones already defined, taking into account that it filters tables or columns. Therefore, it is used to calculate static and non-aggregated values.
  • Measures: only visible through visualizations. They are evaluated through "filter context". Pro tip: they are useful when calculating aggregates.
  • Filter Context: filters the tables in the data model. DAX creates Filter Context when dimensions are added to rows, columns, slicers, and filters in a report. This Filter Context propagates in the model's relationships, from the 1 side to the * side.
  • Row Context: "iterates" the rows of a table. DAX creates Row Context when a Calculate Column is added to the model. This Row Context does not propagate in the model and needs the help of functions like RELATED or RELATEDTABLE.

DAX Engines:

DAX works through the "formula engine" (executes the query) and the "storage engine" (fetches information for the query).

The FE receives a DAX Query and converts it into a Query plan (list of steps) that it sends to the SE to be executed and turned into a data cache that is sent back to the FE to be evaluated and deliver a result.

There are 2 types of SE:

  • Vertipaq: stores the information in the PC's memory (import mode). Vertipaq has a column structure, that is, it saves each column separately, allowing for speed and efficiency.

The objective of "Vertipaq" is to reduce the storage required to evaluate the DAX Query. It does this through "Value Encoding", "Hash Encoding", and "Run Length Encoding" based on cardinality, repeated values, row count, and data type.

Value Encoding: uses mathematical processes that determine the relationship between the values within a column and converts them into smaller values to optimize storage - only works with integers. Eg: City ID (10215,10216) transforms into City ID (215,216).

Hash Encoding and Run Length Encoding in Vertipaq:

Hash Encoding is a technique used in Vertipaq engine to build a dictionary of distinct items in a column and assign a unique number (integer or decimal) to each item in an indexed fashion, based on the number of items instead of the length of the string.

For example, a column of Products (CB, T, B, B, CB, T) would be encoded as (CB=0, T=1, B=2) and then as (0, 1, 3, 3, 0, 1).

Run Length Encoding (RLE) is another technique used to reduce the size of a column by grouping duplicates. Columns can have both RLE and Hash Encoding/Value Encoding (VE) at the same time to further optimize storage.

"Vertipaq Relationship" refers to the relationship between the Vertipaq engine and the data it stores in memory. The engine uses Value Encoding, Hash Encoding, and Run Length Encoding to optimize storage and reduce the storage necessary to evaluate DAX queries.
